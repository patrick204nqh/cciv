# ADR-003: Asset Pipeline Architecture

## Status
Accepted

## Date
2026-06-27

## Context
The codebase currently supports exactly one model (the CCIV ship) via a bespoke pipeline: `pull-reference.mjs` downloads from Poly Haven, `build-model.mjs` extracts geometry into 29 JS literal files committed in `src/models/ship/data/`, and `compile-model.ts` produces a GLB that the runtime never loads. Textures (358 MB of 4K JPGs) are committed in `public/textures/`. Build scripts are split across `scripts/` (Node.js .mjs) and `src/scripts/` (TypeScript .ts), with the latter importing application code — a reverse dependency.

ADR-001 established the Model Abstraction Layer (ModelConfig, ModelEntity, ModelRegistry). ADR-002 established the Scene Entity Lifecycle (SceneEntity, EntityManager, EventBus). Both decisions remain sound — this ADR extends them with a complete asset pipeline.

The project now has three requirements beyond the initial scope:
1. **Multiple providers** — assets come from Poly Haven, Sketchfab, procedural generation, and custom uploads, not one source
2. **Multiple creation paths** — extracted (from external sources), procedural (code-generated geometry), and composite (assembled from primitives + sub-models)
3. **Engine-friendly authoring** — engineers create and iterate on models in TypeScript, not via a UI or by editing raw vertex data

## Decision

### File Structure

```
src/
  primitives/          ← Library of primitive geometry builders (box, cylinder, sphere…)
  generators/          ← Parameterized geometry generators (hull, rigging…)
  models/              ← Model configs, discovered by filesystem convention
    <id>/
      config.ts        → ModelConfig (one of: extracted, procedural, composite)
  worlds/              ← World composition files
    <name>.ts          → WorldConfig
  providers/           ← Asset provider abstraction
    types.ts           → AssetProvider interface
    polyhaven.ts       → Poly Haven adapter
    registry.ts        → ProviderRegistry
  pipeline/            ← Build pipeline (build-time only)
    pull.ts            → Download raw assets from providers → .cache/raw/
    build.ts           → Transform raw/definition → .cache/processed/
    compile.ts         → Compile to GLB (Draco) → public/models/
    publish.ts         → Generate public/models/manifest.json
  loaders/             ← Runtime loaders
    glb-loader.ts      → Load GLB from URL → THREE.Group
    model-loader.ts    → Resolve model ref → apply overrides → ModelEntity
    catalog.ts         → Read manifest.json at runtime
  entity/              ← Existing SceneEntity lifecycle (unchanged)

public/
  models/              ← .glb artifacts + manifest.json (build output, gitignored)
  textures/            ← Generated textures (build output, gitignored)

.cache/
  raw/                 ← Provider downloads (gitignored)
  processed/           ← Intermediate geometry (gitignored)
```

### Pipeline Stages

| Step | Command | Input | Output |
|---|---|---|---|
| `pull` | `npm run pipeline:pull` | provider refs in `config.ts` | `.cache/raw/<id>/` |
| `build` | `npm run pipeline:build` | `src/models/<id>/config.ts` | `.cache/processed/<id>/` |
| `compile` | `npm run pipeline:compile` | processed dirs | `public/models/<id>.glb` (Draco) |
| `publish` | `npm run pipeline:publish` | all .glb + metadata | `public/models/manifest.json` |

A single `npm run build:models` runs all four steps.

### Model Creation Paths

All model configs live in `src/models/<id>/config.ts`, discovered automatically by filesystem convention — no central registry needed.

**Extracted** — for assets from external providers:
```typescript
// src/models/ship/config.ts
export default {
  type: 'extracted',
  provider: 'polyhaven',
  asset: 'ship_pinnace',
  transform: { scale: 2.7 },
  materialOverrides: { hull: { color: 0x1c160e } },
} satisfies ExtractedModelConfig;
```

**Procedural** — for code-generated geometry:
```typescript
// src/models/buoy/config.ts
import { generateBuoy } from '../../generators/buoy';
export default {
  type: 'procedural',
  generator: generateBuoy,
  params: { height: 3, radius: 0.8 },
  material: { color: 0xff4422, roughness: 0.6 },
} satisfies ProceduralModelConfig;
```

**Composite** — for models assembled from primitives and sub-models:
```typescript
// src/models/lighthouse/config.ts
export default {
  type: 'composite',
  parts: [
    { primitive: 'cylinder', params: { radius: 2, height: 20 }, at: [0, 10, 0] },
    { primitive: 'sphere', params: { radius: 1.5 }, at: [0, 21, 0] },
    { model: 'buoy', at: [0, 0, 0] },
  ],
} satisfies CompositeModelConfig;
```

### Provider Seam

```typescript
interface AssetProvider {
  readonly id: string;
  pull(assetId: string, dest: string): Promise<AssetBundle>;
}

interface AssetBundle {
  meshes: Array<{ pos: Float32Array; nml: Float32Array; uv: Float32Array; indices: Uint16Array }>;
  textures: Map<string, string>;  // map key → file path
  metadata: Record<string, unknown>;
}
```

Providers are registered in a `ProviderRegistry`. Built-in providers:
- `polyhaven` — downloads from Poly Haven API
- `local` — reads from local filesystem path

### Primitives Library

```typescript
// All primitives share a common signature
type PrimitiveFn = (params: Record<string, number>) => THREE.BufferGeometry;

// Built-in set
buildBox(w: number, h: number, d: number): BufferGeometry
buildCylinder(rTop: number, rBot: number, height: number, segments?: number): BufferGeometry
buildSphere(radius: number, segments?: number): BufferGeometry
buildPlane(w: number, h: number, segW?: number, segH?: number): BufferGeometry
buildLathe(points: Vector2[], segments?: number): BufferGeometry
```

### World Composition

```typescript
// src/worlds/north-sea.ts
export const northSea: WorldConfig = {
  id: 'north-sea',
  models: [
    { ref: 'ship', at: [0, 0, 0], scale: 2.7 },
    { ref: 'buoy', at: [50, 0, 30] },
    { ref: 'buoy', at: [-50, 0, -20] },
    { ref: 'lighthouse', at: [-120, 0, -60] },
    { ref: 'island', at: [200, 0, 100] },
  ],
  environment: {
    ocean: true,
    sky: true,
    lighting: 'day',
  },
};
```

Worlds are loaded at startup by a `WorldLoader`:
```typescript
// Runtime flow
const world = northSea;
const loader = new WorldLoader(catalog, modelLoader);
await loader.load(world, scene);
```

### Runtime Loading

The runtime never imports geometry data files. It reads `manifest.json` (embedded in the Vite build or fetched at startup):

```json
{
  "ship": { "glb": "/models/ship.glb", "provider": "polyhaven", "polyCount": 168317 },
  "buoy": { "glb": "/models/buoy.glb", "provider": "procedural" },
  "lighthouse": { "glb": "/models/lighthouse.glb", "provider": "composite" }
}
```

`ModelLoader.load(ref)` looks up `ref` in the catalog, fetches the GLB, applies material overrides, wraps in a `ModelEntity`, and caches it in memory. Subsequent loads of the same ref return the cached entity.

### Migration Path

The existing CCIV ship is migrated as follows:
1. `src/models/ship/config.ts` becomes the `extracted` type with `provider: 'polyhaven'`
2. `src/models/ship/data/` files are deleted (they live in `.cache/` now)
3. `src/models/ship/index.ts` is deleted (it was only needed for the in-code pipeline)
4. `public/textures/ship/` is gitignored; pulled at build time
5. `src/scripts/compile-model.ts` moves to `scripts/compile-model.ts` and is rewritten to use the provider's output directory
6. `entity/ship-entity.ts` stays unchanged — it receives a `ModelEntity`, same as before

## Alternatives Considered

### Central model registry (models.json)
Rejected in favor of filesystem convention. Adding a model = creating a directory. No config to update, no merge conflicts.

### JSON-only for composite models
Rejected — TypeScript is the primary authoring path for engineer-friendly workflows. JSON can be added later as a secondary format for user-generated content.

### Single monolithic pipeline script
Rejected in favor of composable stages (pull/build/compile/publish). Each stage can be run independently for debugging. A future orchestrator (`build:models`) chains them.

### Keep build scripts in scripts/ + src/scripts/
Rejected. All pipeline code moves to `scripts/pipeline/` with shared types. No reverse dependency (pipeline → app code). The manifest.json format is the shared contract.

### Runtime GLB download from CDN
Rejected for now — GLBs live in `public/models/` and are served by Vite. A future CDN layer is just a URL prefix change.

## Consequences
- **Positive**: Adding a new model = `mkdir src/models/<id>/ && write config.ts` + optional generator function. No data files committed, no config registration.
- **Positive**: 18 MB of JS literal geometry and 358 MB of textures removed from git. Repo size drops to ~2 MB.
- **Positive**: The provider seam enables any asset source. Future providers (Sketchfab API, Unity Asset Store, DCC tool exports) are adapters behind the same interface.
- **Positive**: Worlds decouple scene composition from entity logic. A world is pure data (model refs + positions) — testable, shareable, and buildable without touching render code.
- **Positive**: Primitives and generators give engineers a code-native path to create geometry. No Blender, no GLB export, no vertex data wrangling.
- **Neutral**: Existing code must be migrated. The ship entity's interface doesn't change — `createShipEntity(model)` still works — but where `model` comes from changes (GLB loader vs in-code data).
- **Neutral**: Build time increases (Draco compression is CPU-intensive). Mitigated by caching — only changed models recompile.
- **Negative**: Async loading adds complexity. `main.ts` must await `WorldLoader.load()` before starting the render loop. Mitigated by showing a loading state or splash.

## ADR References
- Extends ADR-001 (Model abstraction layer) — the ModelConfig/ModelEntity/ModelRegistry seam stays, but the data source moves from in-code JS files to compiled GLB artifacts
- Extends ADR-002 (Scene entity lifecycle) — entity wiring moves from inline `attach()` calls to declarative WorldConfig
- Supersedes ADR-001's "flat data files" format — data files become build intermediates in `.cache/` rather than committed source
- Supersedes ADR-001's "singleton registry" is unchanged — ModelRegistry remains a singleton
- Supersedes ADR-002's "thin main.ts" — main.ts becomes a WorldConfig loader instead of a series of attach() calls

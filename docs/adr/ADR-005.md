# ADR-005: EntityManager and Kernel Lifecycle Refinement

## Status
Accepted

## Date
2026-06-27

## Context
ADR-002 established the EntityManager with a `SceneEntity` lifecycle (`onAttach`, `onBeforeUpdate`, `onUpdate`, `onDetach`) and stated that the EntityManager "runs the requestAnimationFrame loop." In practice, two mismatches emerged:

1. **EntityManager does not own the RAF loop.** The `Kernel` class (`src/kernel.ts`) runs the render loop and delegates entity updates via a `simulationPlugin` that calls `entityManager.update(dt)`. This plugin only activates in 'play' mode. In 'edit' mode, entities are frozen — no `onUpdate` calls, no wave displacement, no ship physics. ADR-002's statement is stale.

2. **No explicit pause control.** Whether entities update depends on which plugins are active. The edit/play mode switch propagates through the plugin system's mode filtering — an indirect, non-obvious coupling. Adding a system that runs in both modes but shouldn't update entities is currently impossible without editing the simulation plugin.

3. **Resource cleanup is ad-hoc.** Every entity creates its own `Disposer` and manually registers Three.js objects. There is no enforcement — nothing prevents a developer from creating a `BufferGeometry` and forgetting to register it with the disposer. Entity lifecycle (attach → detach) and resource cleanup are separate concerns with no formal relationship.

## Decision

### 1. EntityManager owns per-entity Disposers

EntityManager creates and owns a `Disposer` for each attached entity. The `SceneEntity.onAttach` signature gains an optional `disposer` parameter:

```typescript
interface SceneEntity {
  readonly id: string;
  onAttach(scene: THREE.Scene, disposer?: Disposer): void;
  onBeforeUpdate?(dt: number): void;
  onUpdate?(dt: number): void;
  onDetach(): void;
}
```

Entities that need cleanup (all current entities) register Three.js objects with the provided Disposer instead of creating their own. When `entityManager.detach(entity)` is called, the EntityManager disposes the associated Disposer, guaranteeing cleanup. If an entity doesn't need the Disposer (pure-logic entities), it simply ignores the parameter.

### 2. EntityManager has explicit pause control

```typescript
class EntityManager {
  setPaused(paused: boolean): void;
  // update() is a no-op when paused
}
```

`update(dt)` checks the pause flag before iterating entities. The simulation plugin always calls `entityManager.update(dt)` regardless of mode — the pause flag is the sole gate.

### 3. Kernel.setMode() propagates pause

The `mode` setter is replaced with an explicit `setMode(m)` method:

```typescript
class Kernel {
  setMode(m: 'edit' | 'play'): void {
    if (this._mode === m) return;
    this._mode = m;
    entityManager.setPaused(m === 'edit');
    for (const p of this.registry.getAll()) {
      p.onModeSwitch?.(prev, m);
    }
  }
}
```

This makes the pause signal explicit and traceable. Plugins still receive `onModeSwitch` for their own mode-dependent behavior (e.g., gizmos visibility), but they no longer gate entity updates.

### 4. RAF loop stays in Kernel

We explicitly accept the deviation from ADR-002: the Kernel owns the RAF loop. The EntityManager is a passive participant — it provides `update(dt)` and manages lifecycle, but doesn't call `requestAnimationFrame`. This is documented in the domain model so future engineers don't re-debate it.

## Alternatives Considered

### EntityManager owns the RAF loop (as ADR-002 stated)
Rejected. The Kernel needs to interleave render, control updates, and plugin rendering with entity updates. Two `requestAnimationFrame` loops would create a coordination problem (which runs first? how do they share dt?). One loop with explicit ordering is simpler.

### WeakRef-based leak detection
Rejected as too speculative. The FinalizationRegistry approach would add complexity for a theoretical problem. If resource leaks are observed, revisit this.

### Dependency-injected Disposer
Passing a Disposer factory to each entity at construction time instead of through onAttach. Rejected because entities are created by factory functions, not constructors — the Disposer doesn't exist until attach. The onAttach parameter is honest about when the Disposer becomes available.

## Consequences
- **Positive**: Entity cleanup is guaranteed and centralized. No more per-entity `disp` variable pattern.
- **Positive**: Pause semantics are explicit at the EntityManager level. Edit/play mode switching is traceable through `kernel.setMode()`.
- **Positive**: All existing entities continue to work — they accept the optional disposer parameter and register with it.
- **Positive**: Future systems (e.g., a minimap or debug overlay that runs in both modes) can use EntityManager without being frozen in edit mode.
- **Neutral**: Existing entity tests need minor updates if they assert on the exact shape of `onAttach`. The disposer parameter is optional, so existing mock entities are unaffected.
- **Negative**: The Disposer is now shared between EntityManager and the entity. An entity that calls `disposer.dispose()` directly would break EntityManager's ability to clean up. Mitigated by documentation and the fact that all current entities call `disposer.dispose()` only in `onDetach`, which EntityManager calls before disposing.

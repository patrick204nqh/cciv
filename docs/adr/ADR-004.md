# ADR-004: WorldConfig Unification

## Status
Accepted

## Date
2026-06-27

## Context
ADR-003 established a `WorldConfig` type in `src/worlds/types.ts` for declarative scene composition alongside a `WorldLoader` that resolves model refs. Separately, a `LocationPreset` type in `src/state/types.ts` evolved with richer data (environment state, per-instance material overrides) and became the de facto world definition used by the store and location switcher.

This created two problems:
1. **Two shapes, same concept** — `WorldConfig` (marked `@deprecated`) and `LocationPreset` describe the same thing with different structures. Engineers must guess which one to edit when adding a buoy to a world.
2. **Partial loading** — `main.ts` loaded both: `WorldConfig` from `north-sea.ts` for the ship entity, and `LOCATION_PRESETS` from `state/worlds.ts` for the instance manager. The `WorldLoader` ignored its own environment config entirely — entities were still hardcoded in `main.ts`.
3. **No entity-creation seam** — Adding a new entity type (e.g., a fog entity) required editing `main.ts`. A world config couldn't express "what entities exist."

## Decision

### 1. Unify on the `LocationPreset` shape

Adopt the `LocationPreset` structure as the single `WorldConfig` type. It already contains everything a world needs: environment state, model instances with transforms, and material overrides. Rename it to `WorldConfig` and delete the old `@deprecated` type.

### 2. Add a `behavior` field to model instances

```typescript
interface WorldConfig {
  id: string;
  environment: EnvironmentState;
  instances: Record<string, InstanceDef>;
}

interface InstanceDef {
  ref: string;
  transform: { position, rotation, scale };
  visible: boolean;
  behavior?: 'vessel';           // new
  materials?: Record<string, MaterialOverride>;
}
```

Instances without a `behavior` are static (placed in the scene, no per-frame logic). `behavior: 'vessel'` enables wave-response physics, event emission, and automatic spray/wake.

### 3. WorldLoader creates all entities

`WorldLoader.load(config)` returns `SceneEntity[]`:
- One `SceneEntity` per model instance (static or `vessel`)
- One ocean entity (if config.environment.ocean)
- One sky entity
- One lighting entity
- Spray and wake entities for each `vessel` instance

`main.ts` becomes a single attach loop. No hardcoded entity factories.

### 4. Delete dead types

- `src/worlds/types.ts` — delete `WorldConfig`, `ModelInstance`, `EnvironmentConfig` (the `@deprecated` ones)
- `src/worlds/north-sea.ts` — delete (data lives in `state/worlds.ts`)
- `src/textures/manifest.json` — delete (textures baked into GLB)
- `src/material/types.ts` — delete (MaterialSpec unused; materials in GLB)
- `src/material/` — delete directory (MaterialRegistry never existed; types.ts is the only file)

## Alternatives Considered

### Design a new shape from scratch
Rejected. The `LocationPreset` already works and has been tested across four location presets. Redesigning would introduce risk for no gain.

### Keep both shapes, add a converter
Rejected. Two shapes for the same concept is the source of the friction. A converter hides the duplication rather than removing it.

### Make WorldLoader return WorldLoadResult with typed entity refs
Rejected. `SceneEntity[]` is simpler and more composable. The caller (main.ts or a test) just iterates and attaches.

## Consequences
- **Positive**: One WorldConfig shape, one loader, one source of truth for scene composition.
- **Positive**: Adding a new entity type = one factory call inside WorldLoader. No main.ts edits.
- **Positive**: Adding a second vessel = `behavior: 'vessel'` on an instance. Spray and wake follow automatically.
- **Positive**: `isClaimed('ship')` hack in instance-manager.ts is deleted — all instances come from one config.
- **Neutral**: Existing `north-sea.ts` and `state/worlds.ts` must be merged. The `north-sea.ts` model positions and environment are duplicated; the merge picks `state/worlds.ts` as canonical.
- **Negative**: `WorldLoader` becomes deeper (more behaviour behind the same interface). It now needs to know about entity factories (ocean, sky, lighting, vessel), creating a new dependency. Acceptable — these factories were already scattered across `main.ts`.

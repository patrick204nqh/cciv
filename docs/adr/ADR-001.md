# ADR-001: Model Abstraction Layer

## Status
Accepted

## Date
2026-06-26

## Context
The codebase currently has exactly one 3D model (the CCIV ship) with a bespoke pipeline: a one-shot extraction script (`scripts/extract-cciv.mjs`), 28 hardcoded geometry data files in `src/ship/cciv/`, 7 per-group texture wrapper functions in `src/textures/index.ts`, and inline material construction in `src/ship/cciv/ship.ts`. The caller interface is `createShip(): THREE.Group`.

The project plans to scale from one ship to a "dream world" with many models. Models may come from three sources:

1. **Extracted** — imported from glTF/GLB once, geometry extracted to code (owned, customizable)
2. **External** — GLB/glTF loaded at runtime from URL or local file (no extraction step)
3. **Procedural** — built entirely in code with geometry helpers

Without a shared seam, each new model would require replicating the entire CCIV pipeline — extraction script, data files, texture wrappers, material setup. This is a shallow pattern: the interface is nearly as complex as the implementation.

## Decision
Introduce a **Model Abstraction Layer** with the following shape:

### Concepts

- **ModelConfig** — configuration data that describes a model. Discriminated by source type (`extracted`, `procedural`, `external`), each with its own shape.
- **ModelEntity** — the runtime object returned to callers. A `THREE.Group` with metadata and lifecycle methods.
- **ModelRegistry** (singleton) — central registry that tracks all active models. Handles disposal coordination and global queries.

### Principles

- **Same seam for all sources**: extracted, procedural, and external models all produce an identical `ModelEntity`. Callers never know the source type.
- **Config-driven**: mesh-group-to-texture mapping lives in config data, not in adapter classes. Adding a model model = writing a config entry.
- **Auto transforms**: default scale, rotation, and offset are applied automatically from config.
- **Flat data files**: extracted geometry uses one file per attribute array (current CCIV convention preserved).
- **Registry lifecycle**: `dispose()` disposes Three.js resources AND unregisters from the singleton `ModelRegistry`.
- **Singleton registry**: no dependency injection — the `ModelRegistry` is globally accessible.

### Pipeline

- **Build-time extraction**: a reusable Node script (generalizing `extract-cciv.mjs`) accepts a glTF/GLB and model ID, emits `_pos.js`, `_nml.js`, etc. into `src/models/<id>/`.
- **Factory routing**: a factory function accepts `ModelConfig` and dispatches to the correct adapter based on source type.

## Alternatives Considered

### Adapter-class-per-model
Each model type has a small class that knows its own mesh groups and texture mappings. Rejected in favor of config data because it scales better (adding a model = editing a config, not writing code) and is more AI-navigable (configs are declarative).

### Runtime glTF loading only (no extraction)
Avoids the extraction step entirely; all models loaded at runtime. Rejected because it makes the codebase depend on external files that may change or disappear, and prevents customizing geometry.

### Dependency-injected registry
Passing a registry instance around instead of using a singleton. Rejected because this project has a single renderer/scene and no testing of multiple registries simultaneously — the singleton constraint is honest.

### Bundled extraction format
One JSON blob per model instead of flat `_pos.js` files. Rejected to preserve the current convention, which works well with tree-shaking and diff review.

## Consequences
- **Positive**: Adding a new model = write a config entry + (for extracted) run extraction script. No new wrapper functions, no new adapter classes.
- **Positive**: Testability improves — a model can be constructed with a config and its `ModelEntity` inspected without running the renderer.
- **Positive**: Model metadata (license, source, poly count) travels with the entity, enabling attribution and debugging.
- **Neutral**: The CCIV ship must be retrofitted to the new interface. This is mechanical — the existing data files and ship.ts become one config entry.
- **Negative**: Introduces a registry singleton, which adds a global dependency. Acceptable given the single-scene constraint.

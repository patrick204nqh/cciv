# ADR-002: Scene Entity Lifecycle

## Status
Accepted

## Date
2026-06-26

## Context
The animation loop in `src/main.ts:49-77` is a single monolithic procedure that manually orchestrates every scene entity — it updates the ocean, samples waves, positions the ship, positions the wake, updates spray particles, and renders. Every entity's update logic is interleaved with no seam between them.

Adding a second vessel, an island, or any new entity requires editing the loop body and understanding all existing steps. The loop is untestable — you cannot verify ship wave response without running the full render pipeline.

Previous architecture review (see `docs/adr/ADR-001`) established a `ModelEntity` interface for model representation. This ADR addresses the runtime orchestration layer.

## Decision
Introduce an **EntityManager** with a `SceneEntity` interface:

### SceneEntity interface
```typescript
interface SceneEntity {
  readonly id: string;
  onAttach(scene: THREE.Scene): void;
  onBeforeUpdate(dt: number): void;  // marks dirty state for events
  onUpdate(dt: number): void;        // core per-frame logic
  onDetach(): void;
}
```

### EntityManager
- Singleton (same pattern as `ModelRegistry`)
- Owns the entity list (`Set<SceneEntity>`)
- Runs the `requestAnimationFrame` loop
- Provides `attach(entity)`, `detach(entity)`, `detachAll()`
- Calls lifecycle hooks in order: `onBeforeUpdate` → events → `onUpdate` for all entities each frame
- Testable: `manager.update(dt)` can be called directly without a browser

### Event bus
- Singleton event bus alongside EntityManager
- Entities fire events when their state changes (e.g., `entity:position-changed` after `onBeforeUpdate` marks dirty position)
- Other entities listen without direct coupling
- Minimal event set:
  - `entity:attached` — entity added to manager
  - `entity:detached` — entity removed from manager
  - `entity:position-changed` — entity's position/quaternion changed this frame

### Principles
- **Self-contained entities**: entities import what they need directly (waves, materials, registry). No `worldContext` object.
- **Decoupled via events**: entities that depend on another entity's state (spray needs ship position) use the event bus, not direct references.
- **Thin main.ts**: entry point creates manager, attaches initial entities, calls `manager.start()`. No per-entity logic in the loop.

## Alternatives Considered

### Keep all logic in main.ts
Rejected. Does not scale beyond one ship. The loop is untestable and every new entity requires editing it.

### Inject worldContext with shared state
Rejected in favor of self-contained entities that import dependencies directly. Simpler, fewer abstractions, more transparent.

### Direct entity references (spray receives ship reference)
Rejected in favor of event bus to keep entities decoupled. Spray doesn't need to know *which* entity it follows — it listens for any position-changed event from entities tagged as "vessel."

## Consequences
- **Positive**: Adding a new entity = write a class + attach to manager. No changes to main.ts or other entities.
- **Positive**: The update loop becomes testable — `entityManager.update(0.016)` in a unit test with no DOM or renderer.
- **Positive**: Event bus enables emergent behaviours (spray follows any vessel without knowing which one).
- **Neutral**: Existing entities (ocean, wake, spray) must be refactored into `SceneEntity` implementations. This is mechanical — their current `updateOcean`/`updateSpray` functions become `onUpdate` methods.
- **Negative**: Event bus adds an indirection layer. Debugging event chains is harder than direct calls. Mitigated by keeping the event set minimal (3 events).
